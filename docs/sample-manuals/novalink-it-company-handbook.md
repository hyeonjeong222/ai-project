# NovaLink Labs 사내 업무 매뉴얼

> **문서 구분:** 전사 공통·부서별 온보딩 매뉴얼  
> **대상:** 신규 입사자, 계약직·인턴, 협업 부서 구성원  
> **문서 버전:** v1.0 / 2026-07-13  
> **문서 오너:** People Operations(피플옵스) / 운영정책팀  
> **적용 범위:** NovaLink Labs 및 승인된 외부 협력사

## 1. 회사와 이 매뉴얼의 목적

### 1.1 회사 소개

**NovaLink Labs(노바링크 랩스)**는 기업이 고객 문의, 업무 문서, 반복 업무를 더 빠르게 처리할 수 있도록 B2B SaaS(서비스형 소프트웨어)를 만드는 가상의 IT 회사입니다. 핵심 제품은 다음과 같습니다.

- **NovaDesk:** 고객 문의와 상담 이력을 관리하는 고객지원 플랫폼
- **NovaFlow:** 사내 승인·알림·반복 업무를 자동화하는 워크플로 제품
- **NovaInsight:** 제품 및 고객 데이터를 시각화하는 분석 서비스

우리의 원칙은 “고객의 업무 시간을 줄이는 기능은 빠르게, 고객의 신뢰를 잃을 수 있는 변경은 신중하게”입니다.

### 1.2 매뉴얼 사용법

이 문서는 업무 판단을 돕는 기본 기준입니다. 모든 상황을 대신 결정하지는 않습니다.

1. 먼저 해당 업무의 **공통 지침**과 **소속 부서 지침**을 확인합니다.
2. 고객 정보, 비용, 권한, 외부 공개가 포함되면 승인 절차를 한 단계 더 확인합니다.
3. 문서에 없는 상황은 추측해 처리하지 말고, 업무 요청 티켓에 질문을 남기거나 담당자에게 문의합니다.
4. 긴급 장애·보안 사고는 일반 승인보다 **사고 대응 절차**를 우선합니다.

### 1.3 역할과 의사결정 원칙

| 구분 | 할 수 있는 일 | 사전 승인 필요 예시 |
| --- | --- | --- |
| 구성원 | 맡은 업무 수행, 개선 제안, 내부 문서 작성 | 고객 데이터 내려받기, 외부 발표, 비용 집행 |
| 담당자(DRI) | 업무 결과물의 최종 품질 확인 | 정책 변경, 서비스 영향 변경 |
| 팀 리드 | 팀 우선순위·리소스 결정 | 채용 확정, 예산 초과, 고위험 릴리스 |
| 부서장 | 부서 간 갈등 조정, 운영 책임 | 계약·가격·보안 예외 승인 |
| 경영진 | 회사 전략 및 주요 리스크 결정 | 법적 분쟁, 대외 공식 입장 |

**DRI(Directly Responsible Individual)**는 업무를 실제로 끝까지 책임지는 한 명의 담당자입니다. 여러 명이 협업해도 DRI는 반드시 한 명을 지정합니다.

## 2. 전사 공통 업무 지침

### 2.1 근무와 소통

- **핵심 협업 시간:** 평일 10:00~16:00(KST). 이 시간에는 연락 가능 상태를 유지합니다.
- **비동기 소통:** 긴 설명·결정 기록은 Slack 대신 Notion 또는 Jira 티켓에 남깁니다. Slack은 빠른 확인과 논의용입니다.
- **응답 기준:** 일반 내부 문의는 영업일 1일 내, 고객 영향 장애는 즉시 확인 후 15분 내 최초 상태를 공유합니다.
- **회의 원칙:** 목적, 의사결정이 필요한 항목, 사전 읽을 자료를 초대장에 적습니다. 회의 후에는 결정·담당자·기한을 기록합니다.
- **상태 표기:** 자리를 비우거나 업무가 지연될 때는 캘린더와 Slack 상태를 업데이트합니다. 민감한 개인 사정은 공개할 필요가 없습니다.

### 2.2 업무 요청과 우선순위

모든 반복 가능하거나 추적이 필요한 업무는 Jira 티켓으로 관리합니다. 구두 요청만 받고 시작하지 않습니다.

| 우선순위 | 의미 | 최초 대응 / 처리 목표 |
| --- | --- | --- |
| P0 | 전체 서비스 중단, 데이터 유출 의심, 법적 위험 | 즉시 / 사고 대응 절차 |
| P1 | 핵심 기능 장애 또는 다수 고객 영향 | 15분 / 당일 완화 |
| P2 | 일부 고객·기능 영향, 우회 가능 | 영업일 1일 / 계획 수립 |
| P3 | 개선, 문의, 일반 요청 | 영업일 3일 / 백로그 반영 |

티켓에는 최소한 다음을 포함합니다: 배경, 원하는 결과, 영향 사용자, 완료 기준, 기한, 관련 링크, DRI. 완료 기준이 불분명하면 작업 시작 전에 요청자와 합의합니다.

### 2.3 문서 작성 표준

- 제목에는 목적이 드러나는 동사를 사용합니다. 예: `고객 CSV 내보내기 권한 정책`.
- 문서 첫머리에 오너, 최종 수정일, 적용 범위, 관련 링크를 적습니다.
- 사실·정책·결정과 개인 의견을 구분합니다.
- 고객 또는 개인정보는 원문을 붙여 넣지 않습니다. 필요한 경우 마스킹하거나 테스트 데이터를 사용합니다.
- 정책 변경 시 기존 문서의 내용을 조용히 덮어쓰지 말고, 변경 이력과 적용일을 남깁니다.

### 2.4 회의와 의사결정 기록

다음 결정은 반드시 Decision Log(의사결정 기록)에 남깁니다: 제품 범위 변경, 요금·계약 조건, 보안 예외, 고객에게 약속한 일정, 데이터 보존 정책, 서비스 중단을 수반하는 변경.

기록 형식은 `결정 내용 / 배경 / 고려한 대안 / 결정자 / 적용일 / 재검토일`입니다. 반대 의견이 있었던 경우에도 요약해 남깁니다.

## 3. 정보보안·개인정보·접근권한

### 3.1 반드시 지켜야 할 보안 규칙

1. 회사 계정은 개인과 공유하지 않습니다. 공동 계정이 필요하면 시스템 오너에게 역할 기반 계정을 요청합니다.
2. 비밀번호는 비밀번호 관리 도구에만 저장하고, Slack·이메일·문서에 쓰지 않습니다.
3. MFA(다중 인증)는 회사 계정 전체에 사용합니다.
4. 고객 데이터·소스 코드·계약서는 승인된 회사 저장소에서만 처리합니다.
5. 개인 메일, 개인 클라우드, 개인 메신저로 회사 파일을 전송하지 않습니다.
6. 의심스러운 링크·첨부파일·로그인 요청은 클릭하지 말고 보안 채널에 신고합니다.

### 3.2 데이터 등급

| 등급 | 예시 | 저장·공유 기준 |
| --- | --- | --- |
| 공개(Public) | 공개된 블로그, 채용 공고 | 회사 승인 채널에서 외부 공유 가능 |
| 내부(Internal) | 업무 일정, 일반 운영 문서 | 회사 계정 사용자에게만 공유 |
| 기밀(Confidential) | 고객 계약, 미공개 로드맵, 재무 자료 | 업무상 필요한 인원에게만 접근 |
| 제한(Restricted) | 개인정보, API 비밀키, 인증 토큰 | 최소 권한·암호화·접근 기록 필수 |

개인정보에는 이름, 연락처, 이메일, IP, 기기 식별자, 고객 상담 내용 등 개인을 식별하거나 결합해 식별할 수 있는 정보가 포함됩니다. “테스트용”이라도 실제 고객 정보를 사용하면 제한 등급으로 취급합니다.

### 3.3 접근권한 요청·회수

- 새 도구 권한은 Jira의 `Access Request` 양식으로 요청합니다.
- 요청에는 시스템명, 필요한 역할, 업무 목적, 종료 예정일(임시 권한인 경우)을 작성합니다.
- 관리자는 **최소 권한 원칙**으로 승인합니다. 읽기 권한으로 충분하면 쓰기·관리자 권한을 주지 않습니다.
- 퇴사·직무 변경 시 리드는 당일 권한 회수 목록을 피플옵스와 IT 운영팀에 전달합니다.
- 90일 이상 사용하지 않은 임시 권한은 정기 검토에서 회수할 수 있습니다.

### 3.4 보안 사고 신고

다음은 즉시 신고 대상입니다: 비밀키 노출, 잘못된 수신자에게 파일 전송, 계정 탈취 의심, 악성코드 의심, 고객 데이터의 비정상 조회·다운로드, 권한 없는 시스템 변경.

신고 시에는 **무엇을, 언제, 어디서 발견했는지**만 사실대로 전달합니다. 스스로 로그를 지우거나 관련 메시지를 삭제하지 않습니다. 보안 담당자는 원인 분석·격리·고객 통지를 결정합니다.

## 4. 제품·엔지니어링 부서 지침

### 4.1 개발 시작 전

- Jira 티켓에 사용자 문제, 성공 지표, 완료 기준, 예외 상황을 확인합니다.
- API·데이터베이스·권한에 영향을 주면 설계 문서(RFC 또는 ADR)를 작성합니다.
- 고객에게 보이는 동작 변경은 Product Manager(PM), Design, QA와 사전에 범위를 합의합니다.
- 기존 기능을 대체하는 경우 롤백 방법과 데이터 마이그레이션 계획을 먼저 정합니다.

### 4.2 코드와 리뷰

- 모든 코드 변경은 Pull Request(PR)로 제출합니다. `main` 브랜치에 직접 푸시하지 않습니다.
- PR에는 문제, 변경 내용, 테스트 방법, 위험 요소, 화면 변경 시 스크린샷을 포함합니다.
- 최소 1명의 승인된 리뷰어가 필요하며, 권한·결제·개인정보·데이터베이스 변경은 해당 오너의 추가 리뷰가 필요합니다.
- 리뷰어는 기능 정확성, 보안, 장애 가능성, 테스트 누락, 유지보수성을 확인합니다. 취향 차이는 팀 코딩 규칙을 우선합니다.
- 비밀키, 실제 고객 데이터, 대용량 생성 파일은 저장소에 커밋하지 않습니다.

### 4.3 테스트 기준

| 변경 유형 | 필수 확인 |
| --- | --- |
| 화면·일반 로직 | 단위 테스트 또는 수동 시나리오, 린트·타입 검사 |
| API 변경 | 정상·오류·권한 없음·입력 검증 시나리오 |
| DB 스키마 | 마이그레이션, 롤백 가능성, 기존 데이터 영향 |
| 결제·권한·개인정보 | 자동 테스트, 최소 두 명의 검토, 스테이징 검증 |
| 장애 수정 | 재현 시나리오와 재발 방지 테스트 |

### 4.4 배포와 롤백

- Production 배포 전 CI(자동 검사)가 통과해야 합니다.
- 영향이 큰 변경은 Feature Flag(기능 플래그) 또는 점진 배포를 사용합니다.
- 배포 담당자는 배포 시간, 변경 목록, 모니터링 항목, 롤백 담당자를 공지합니다.
- 오류율·응답 시간·핵심 전환 지표가 기준치를 넘으면 즉시 롤백 또는 플래그 비활성화를 검토합니다.
- 긴급 배포라도 사후 1영업일 내 PR·테스트·변경 기록을 보완합니다.

### 4.5 장애 대응

1. **탐지:** 알림 또는 고객 제보를 받은 사람이 장애 채널에 시간과 증상을 기록합니다.
2. **분류:** Incident Commander(IC)를 지정하고 P0~P3 우선순위를 판단합니다.
3. **완화:** 근본 원인보다 고객 영향 축소를 우선합니다. 롤백, 트래픽 제한, 기능 플래그를 사용합니다.
4. **소통:** P0/P1은 30분마다 내부 상태를 갱신하고, 고객 공지는 CS·커뮤니케이션 담당과 정합니다.
5. **사후 분석:** P0/P1은 5영업일 안에 블레임리스 포스트모템을 작성합니다. 개인 비난 대신 시스템·과정 개선에 집중합니다.

## 5. 제품·디자인 부서 지침

### 5.1 제품 기획

PM은 기능 목록보다 사용자 문제를 먼저 정의합니다. 기획서에는 대상 사용자, 현재 불편, 해결 가설, 성공 지표, 제외 범위, 출시 후 확인 방법을 작성합니다.

- 고객 요청 하나만으로 즉시 기능을 약속하지 않습니다. 문제의 빈도와 전략 적합성을 확인합니다.
- 개발 시작 후 요구사항이 바뀌면 영향도(일정·품질·비용)를 공개하고 우선순위를 다시 합의합니다.
- “완료”는 개발 완료가 아니라, 사용자에게 배포되고 성공 지표를 확인할 준비가 된 상태를 뜻합니다.

### 5.2 UX·UI 디자인

- 디자인은 공통 Design System을 우선 사용합니다. 새 컴포넌트는 재사용 필요성을 검토합니다.
- 모든 주요 화면에는 빈 상태, 로딩 상태, 오류 상태, 권한 없음 상태를 설계합니다.
- 색상만으로 상태를 전달하지 않고 텍스트·아이콘 등 보조 수단을 함께 사용합니다.
- 버튼·링크·입력 요소는 키보드로 이동·실행 가능해야 하며, 이미지에는 대체 텍스트를 제공합니다.
- 개인정보를 입력받는 화면은 수집 목적과 필수 여부를 명확히 표시합니다.

### 5.3 사용자 리서치

- 인터뷰 전에는 목적, 질문 가설, 참여 동의, 녹화·보관 여부를 안내합니다.
- 관찰 사실과 해석을 분리해 기록합니다.
- 참가자 개인 정보를 리서치 보고서에 불필요하게 포함하지 않습니다.
- 한 명의 강한 의견만으로 정책이나 로드맵을 변경하지 않습니다.

## 6. 고객성공·고객지원 부서 지침

### 6.1 고객 응대 원칙

고객이 이해할 수 있는 언어로 답하고, 모르는 사실을 추정하지 않습니다. 답변은 다음 순서를 권장합니다: 공감 → 현재 확인된 사실 → 다음 조치와 담당자 → 예상 업데이트 시간.

금지 표현: “아마”, “무조건 가능합니다”, “개발팀 탓입니다”, “곧 처리됩니다(기한 없음)”.  
권장 표현: “현재 확인한 범위에서는…”, “확인 후 오늘 16시까지 다시 안내드리겠습니다.”

### 6.2 티켓 처리 기준

| 유형 | 예시 | 처리 기준 |
| --- | --- | --- |
| 장애 | 로그인 불가, 데이터 조회 오류 | 우선순위 분류 후 엔지니어링 에스컬레이션 |
| 사용 문의 | 설정 방법, 기능 위치 | 지식 문서 근거와 함께 안내 |
| 기능 요청 | 새 연동, 화면 개선 | 요청 배경·영향 사용자·대체 방법 수집 |
| 계정·계약 | 사용자 추가, 플랜 문의 | 권한·계약 담당 확인 후 처리 |
| 보안·개인정보 | 데이터 삭제, 의심 활동 | 즉시 보안·개인정보 담당에게 전달 |

### 6.3 에스컬레이션

다음 경우는 혼자 해결하려 하지 말고 에스컬레이션합니다.

- 고객의 서비스가 중단되었거나 다수 고객에게 같은 문제가 발생할 때
- 계약, 환불, 법률, 보안, 개인정보 관련 약속을 요구받을 때
- 고객이 삭제·내보내기·복구를 요청할 때
- 2회 이상 안내했는데 문제 원인이 불명확할 때

에스컬레이션 티켓에는 고객명, 영향도, 재현 방법, 오류 시각(KST), 스크린샷 또는 요청 ID, 이미 안내한 내용을 포함합니다. 고객 비밀번호나 비밀키는 받지 않습니다.

### 6.4 고객 데이터 조회

- 고객 문제 해결에 필요한 최소 범위만 조회합니다.
- 조회 목적과 티켓 번호를 접근 기록에 남깁니다.
- 화면을 캡처해야 하면 개인정보를 마스킹합니다.
- 고객 환경에서 설정을 변경하기 전에는 변경 내용과 영향 범위를 설명하고 동의를 받습니다. 긴급 장애 완화는 예외로 하되 사후 기록합니다.

## 7. 영업·마케팅 부서 지침

### 7.1 영업 활동

- CRM에는 고객사명, 담당자, 논의 요약, 다음 행동, 예상 금액, 동의 여부를 최신 상태로 기록합니다.
- 제품에 아직 없는 기능·인증·연동·납기일을 확정적으로 약속하지 않습니다.
- 가격 할인, 무료 기간 연장, 맞춤 개발, SLA(Service Level Agreement)는 승인된 가격 정책과 계약 절차를 따릅니다.
- 잠재 고객의 정보를 수집할 때는 목적과 수신 거부 방법을 제공합니다.

### 7.2 제안서와 데모

- 데모는 승인된 데모 환경·가상 데이터로 진행합니다.
- 타 고객의 화면, 데이터, 계약 조건을 예시로 보여주지 않습니다.
- 보안·성능 수치는 측정 기준과 기간을 명시합니다. 측정하지 않은 수치를 만들지 않습니다.
- 제안서의 제품 범위, 지원 범위, 제외 항목은 계약서 초안과 일치해야 합니다.

### 7.3 마케팅 콘텐츠

- 고객 로고·사례·인터뷰는 서면 사용 승인을 받은 뒤 게시합니다.
- 비교 광고는 검증 가능한 공개 자료를 사용하고 경쟁사를 비방하지 않습니다.
- 이메일·광고 캠페인은 수신 동의와 국가별 규정을 확인합니다.
- 성과 보고는 노출·클릭·전환의 정의와 데이터 기간을 함께 표시합니다.

## 8. 운영·재무·피플옵스 지침

### 8.1 구매와 비용 집행

- 회사 카드·법인 비용은 업무 목적에만 사용합니다.
- 구매 전에는 목적, 금액, 공급사, 갱신일, 데이터 처리 여부를 확인합니다.
- 구독형 SaaS는 오너와 갱신일을 계약 관리대장에 기록합니다.
- 개인 결제 후 비용 정산은 영수증, 업무 목적, 참석자(식대인 경우)를 첨부합니다.
- 승인 한도를 넘는 비용은 결제 전에 승인받습니다. 사후 승인은 원칙적으로 허용하지 않습니다.

### 8.2 채용과 면접

- 평가 기준은 직무 역량과 역할 요건에 한정합니다.
- 출신, 가족, 건강, 정치·종교 등 직무와 무관한 민감한 질문을 하지 않습니다.
- 지원자 이력서와 면접 기록은 채용 시스템에서만 관리합니다.
- 면접 피드백은 면접 후 24시간 내 구체적 관찰 사실을 중심으로 작성합니다.

### 8.3 온보딩과 오프보딩

**입사 전:** 계정·장비·필수 권한·첫 주 일정·버디(Buddy)를 준비합니다.  
**첫 30일:** 역할 목표, 제품 교육, 보안 교육, 부서별 핵심 업무를 확인합니다.  
**퇴사 시:** 계정·장비·문서 오너십·진행 중 업무·외부 연락처를 인수인계하고, 접근권한을 당일 회수합니다.

## 9. AI 사용 지침

### 9.1 허용되는 사용

- 초안 작성, 코드 설명, 회의 요약, 익명화된 테스트 데이터 생성, 반복 업무 자동화 제안
- 회사가 승인한 AI 도구에서만 사용
- 생성 결과를 사람이 검토하고, 외부 전달 전 사실·보안·저작권 문제를 확인

### 9.2 금지 또는 사전 승인 필요 사용

- 고객 개인정보, 비밀키, 비공개 소스 코드, 계약서 원문을 공개형 AI 도구에 입력
- AI 결과만으로 채용·평가·해고·대출·가격 등 사람에게 중대한 영향을 주는 결정
- 고객에게 사람 답변인 것처럼 AI 생성 내용을 검토 없이 발송
- 회사 승인 없이 고객 데이터로 AI 모델을 학습·미세조정

### 9.3 AI 답변 검증 체크리스트

1. 근거 문서가 최신인지 확인한다.
2. 사실과 추측이 섞이지 않았는지 확인한다.
3. 고객·직원 개인정보가 노출되지 않았는지 확인한다.
4. 전문 판단(법률·세무·의료 등)은 담당 전문가 검토가 필요한지 확인한다.
5. 답변할 수 없는 내용이면 모른다고 말하고 담당자에게 연결한다.

## 10. 부서 간 협업 시나리오

### 10.1 고객이 기능을 요청했을 때

1. CS 또는 영업이 요청의 문제·영향 사용자·현재 대안을 기록한다.
2. PM이 제품 전략과 고객 영향도를 검토해 상태를 `검토`, `계획`, `보류`, `제외` 중 하나로 갱신한다.
3. 확정 전에는 고객에게 개발 확정을 약속하지 않는다.
4. 계획된 경우 PM이 목표·범위·출시 기준을 만들고, 엔지니어링·디자인이 견적과 위험을 검토한다.
5. 출시 후 CS가 고객에게 변경 내용을 안내하고 피드백을 수집한다.

### 10.2 고객 데이터 삭제 요청이 들어왔을 때

1. 요청자의 권한과 대상 고객사를 확인한다.
2. 삭제 범위·보존 의무·복구 가능 기간을 개인정보 담당과 확인한다.
3. 승인된 절차와 티켓으로 처리하고, 삭제 완료·제외된 데이터·법적 보존 항목을 고객에게 안내한다.
4. 임의 SQL 실행이나 운영 데이터 직접 삭제는 금지한다.

### 10.3 프로덕션 장애가 발생했을 때

1. 발견자는 장애 채널을 만들고 증상·시간·영향을 공유한다.
2. IC가 역할(기술 대응, 고객 소통, 기록)을 배정한다.
3. 고객 영향 최소화 조치를 먼저 실행한다.
4. 해결 후 고객지원팀이 영향 고객에게 결과와 후속 조치를 안내한다.
5. 사후 분석에서 재발 방지 과제를 만들고 오너와 기한을 지정한다.

## 11. 자주 묻는 질문(FAQ)

### Q. 매뉴얼에 없는 고객 질문을 받았습니다.

추정해 답하지 않습니다. 고객에게 확인 중임을 알리고, 지식 문서·티켓을 검색한 뒤 담당 부서에 에스컬레이션합니다. 답을 받은 뒤에는 재사용 가능한 내용인지 검토해 FAQ 또는 매뉴얼 보완을 요청합니다.

### Q. 실수로 고객 정보가 포함된 파일을 잘못 공유했습니다.

즉시 보안 사고로 신고합니다. 파일을 회수하려고 메시지를 삭제하는 것만으로 끝내지 않습니다. 언제, 누구에게, 어떤 정보가 공유됐는지 보존합니다.

### Q. 긴급한 고객 요청이라 승인 없이 권한을 줘도 되나요?

아니요. 업무 영향이 긴급해도 최소 권한·승인 기록 원칙은 유지합니다. 장애 완화를 위한 예외가 필요하면 IC 또는 권한 오너가 승인하고 사후 기록합니다.

### Q. 다른 부서의 업무가 지연되고 있습니다.

개인에게 반복적으로 재촉하기보다 티켓의 우선순위·기한·의존성을 명확히 하고, 필요하면 각 팀 리드에게 조정을 요청합니다.

| 4 | Jira 로그인 | 본인에게 할당된 온보딩 티켓이 보임 | 팀 리드 |
| 5 | 캘린더 확인 | 첫 주 교육·1:1·팀 회의가 표시됨 | 피플옵스 |
| 6 | 이 매뉴얼 읽기 | 1~3장 및 소속 부서 장을 읽음 | 버디 |
| 7 | 장비 확인 | 노트북 잠금, 업데이트, 디스크 암호화 상태 확인 | IT 운영 담당자 |

### A-1-2. MFA 등록 방법

1. 회사 계정 로그인 화면에서 `다중 인증 설정`을 선택합니다.
2. 회사가 허용한 인증 앱(예: Microsoft Authenticator, Google Authenticator)을 휴대전화에 설치합니다.
3. QR 코드를 인증 앱으로 스캔합니다.
4. 앱에 나타난 6자리 숫자를 회사 로그인 화면에 입력합니다.
5. **복구 코드**가 보이면 비밀번호 관리자에만 저장합니다. 사진으로 찍거나 메신저로 보내지 않습니다.
6. 로그인 후 MFA가 실제로 작동하는지 한 번 로그아웃·로그인하여 확인합니다.

**멈춤 기준:** 휴대전화를 분실했거나 복구 코드를 잃어버렸다면 새 계정을 만들지 않습니다. IT 운영 담당자에게 본인 확인 후 복구를 요청합니다.

### A-1-3. 첫 주의 목표

첫 주의 목표는 많은 일을 끝내는 것이 아니라, 제품·팀·도구·보안 기준을 이해하고 작은 업무 한 건을 안전하게 완료하는 것입니다.

- 월요일: 계정·장비·보안 교육 완료
- 화요일: 제품 데모 시청, 소속 팀 업무 흐름 파악
- 수요일: 버디와 함께 기존 Jira 티켓 하나 읽고 질문 정리
- 목요일: 작은 업무(문서 보완, 테스트, 고객 티켓 분류 등) 수행
- 금요일: 팀 리드와 첫 주 회고, 다음 주 목표 2~3개 합의

## A-2. 모든 업무에 공통으로 쓰는 ‘업무 카드’ 작성법

업무를 받으면 바로 작업하지 말고 아래 항목을 Jira 티켓에 적거나 확인합니다. 이 목록이 비어 있으면 요청자에게 질문합니다.

### A-2-1. 업무 카드 양식

```text
[업무 제목]
무엇을: 
왜 필요한가:
누가 영향을 받는가:
완료 모습(완료 기준):
마감일과 이유:
담당자(DRI):
검토자:
관련 문서/화면/티켓 링크:
```

### A-2-2. 좋은 업무 제목과 나쁜 업무 제목

| 나쁜 제목 | 좋은 제목 | 이유 |
| --- | --- | --- |
| 고객 문의 확인 | A사 관리자 로그인 오류 재현 및 최초 답변 작성 | 대상·문제·결과가 보임 |
| 화면 수정 | NovaDesk 사용자 초대 화면의 오류 문구 수정 | 어디를 왜 바꾸는지 보임 |
| 문서 작성 | 신규 입사자용 Jira 티켓 생성 가이드 작성 | 독자와 목적이 보임 |

### A-2-3. 업무 시작 전 5분 점검

아래 질문 중 하나라도 ‘아니오’이면 시작하기 전에 질문합니다.

1. 내가 무엇을 만들어야 하는지 한 문장으로 설명할 수 있는가?
2. 누가 결과를 확인하는지 아는가?
3. 완료됐다고 판단할 기준이 있는가?
4. 고객 정보·비용·권한·외부 공개가 포함되는지 확인했는가?
5. 오늘 해야 하는 일과 나중에 해도 되는 일을 구분했는가?

## A-3. 업무 상태를 공유하는 방법

### A-3-1. 매일 아침 상태 공유

팀에서 정한 채널 또는 스탠드업 문서에 다음 형식으로 짧게 남깁니다.

```text
어제: [완료한 일 1~3개]
오늘: [오늘 끝낼 일 1~3개]
막힌 점: [없음 / 막힌 이유와 필요한 도움]
```

예시:

```text
어제: A사 로그인 오류 티켓의 재현 정보를 정리했고, 도움말 문서 초안을 작성했습니다.
오늘: 오류 로그를 엔지니어링 팀에 전달하고, 고객에게 1차 안내를 보냅니다.
막힌 점: 오류가 특정 브라우저에서만 발생하는지 확인할 테스트 계정이 필요합니다. @민수님 확인 부탁드립니다.
```

### A-3-2. 지연을 알리는 방법

마감일이 지나기 직전까지 기다리지 않습니다. 지연 가능성을 알게 된 순간, 다음 양식으로 티켓과 요청자에게 알립니다.

```text
[현재 상태] 어느 단계까지 완료했습니다.
[지연 이유] 무엇이 예상과 다른지 사실만 적습니다.
[영향] 원래 마감일과 바뀔 수 있는 범위입니다.
[제안] 새 완료 예상일 또는 범위를 줄이는 방법입니다.
[도움 요청] 결정/자료/담당자가 필요한 경우 적습니다.
```

예시: “사용자 목록의 중복 데이터가 예상보다 많아 정리 기준 확인이 필요합니다. 오늘 배포 대신 내일 오전 배포가 가능합니다. 중복 계정을 자동 병합하지 않고 목록만 제공하는 방식으로 범위를 줄일 수 있습니다. 데이터 오너의 기준 확인을 요청합니다.”

### A-3-3. 완료 보고 방법

```text
완료한 일: 
확인한 방법: 
결과 링크: 
남은 위험 또는 다음 할 일: 
도움이 필요한 부분: 
```

‘완료했습니다’만 쓰지 않습니다. 다른 사람이 결과를 재확인할 수 있도록 링크와 확인 방법을 남깁니다.

## A-4. 고객 문의를 처음 받은 사람이 하는 일

이 절은 고객성공팀뿐 아니라 영업·제품·개발팀이 고객 메시지를 직접 받았을 때도 적용합니다.

### A-4-1. 접수 후 10분 안에 할 일

1. 고객 이름, 회사명, 연락 수단, 문의 시각을 티켓에 기록합니다.
2. 고객이 무엇을 하려 했고, 실제로 무엇이 일어났는지 한 문장으로 정리합니다.
3. 서비스 중단·개인정보·보안·결제와 관련 있는지 확인합니다.
4. 이미 알려진 장애인지 상태 페이지와 장애 채널을 확인합니다.
5. 답을 모르면 “확인 후 언제까지 다시 알려드리겠다”는 1차 답변을 보냅니다.

### A-4-2. 고객에게 꼭 물어봐야 하는 정보

오류·기능 문제일 때 다음 중 필요한 것만 물어봅니다.

- 어떤 기능 또는 화면에서 발생했는가?
- 발생한 날짜와 시간은 언제인가? (가능하면 KST)
- 어떤 순서로 눌렀는가?
- 화면에 보인 오류 문구는 무엇인가?
- 영향을 받는 사용자는 한 명인가, 여러 명인가?
- 어떤 브라우저·기기에서 발생했는가?
- 화면 캡처를 받을 수 있는가? (개인정보·비밀번호·토큰은 가린 뒤 요청)

**절대로 묻지 않는 정보:** 비밀번호, 인증 코드, API 비밀키, 전체 주민번호·카드번호, 다른 고객의 정보.

### A-4-3. 최초 답변 문구

```text
안녕하세요, [고객명]님. NovaLink Labs [담당자명]입니다.

[문제 요약] 때문에 불편을 겪고 계신 점을 확인했습니다.
현재 [확인한 사실]까지 확인했으며, [다음 확인/조치]를 진행하겠습니다.
[다음 업데이트 시각]까지 다시 안내드리겠습니다.

빠른 확인을 위해 가능하시다면 [필요한 정보]를 보내주세요.
비밀번호나 인증 코드는 보내지 않으셔도 됩니다.
```

### A-4-4. 문의 분류표

| 고객 말 | 분류 | 첫 담당 | 다음 행동 |
| --- | --- | --- | --- |
| “로그인이 안 돼요.” | 장애 또는 계정 | CS | 영향 범위·오류 문구 확인, P1 가능성 판단 |
| “직원을 추가하고 싶어요.” | 사용 방법 | CS | 권한·플랜 확인 후 도움말 안내 |
| “이 기능을 만들어 주세요.” | 기능 요청 | CS/영업 | 문제·사용자 수·현재 대안 기록 후 PM 전달 |
| “우리 데이터를 지워 주세요.” | 개인정보 요청 | 개인정보 담당 | 요청자 권한 확인 후 승인 절차 시작 |
| “다른 회사 데이터가 보여요.” | 보안 사고 | 보안 담당/IC | 즉시 P0로 신고, 화면 공유 중지 |
| “결제 금액이 달라요.” | 청구·계약 | 재무/CS | 계약·청구서 확인, 임의 환불 금지 |

### A-4-5. 고객 답변 전 마지막 점검

- 문서·티켓에서 확인한 사실만 말했는가?
- 약속한 일정이 실제로 승인되거나 확인된 일정인가?
- 상대 고객 정보나 내부 링크가 포함되지 않았는가?
- 고객이 다음에 무엇을 하면 되는지 알 수 있는가?
- 담당자 이름과 다음 업데이트 시간이 있는가?

## A-5. 고객 지원 티켓을 끝까지 처리하는 절차

### A-5-1. 티켓 상태 정의

| 상태 | 뜻 | 다음에 해야 할 일 |
| --- | --- | --- |
| 신규 | 아직 사람이 확인하지 않음 | 1차 확인·우선순위 분류 |
| 확인 중 | 원인 또는 필요한 정보를 찾는 중 | 다음 업데이트 시각 설정 |
| 고객 응답 대기 | 고객에게 추가 정보를 요청함 | 3영업일 후 1회 리마인드 |
| 내부 처리 중 | 다른 팀이 원인·해결책을 작업 중 | 담당자·예상일을 티켓에 연결 |
| 해결 확인 | 해결책을 적용했고 고객 확인 중 | 고객에게 확인 요청 |
| 종료 | 고객 확인 또는 충분한 기간 경과 후 종료 | 해결 내용·지식 문서 연결 |

### A-5-2. 티켓 종료 조건

다음 네 가지가 모두 충족되어야 종료합니다.

1. 고객의 원래 질문 또는 문제가 티켓 첫 문장에 남아 있다.
2. 어떤 조치를 했는지 기록되어 있다.
3. 고객에게 결과를 안내했거나, 응답이 없어 종료하는 이유가 적혀 있다.
4. 재발 가능성이 있는 경우 지식 문서·버그 티켓·기능 요청에 연결되어 있다.

### A-5-3. 고객 응답이 없을 때

- 3영업일 후: “확인을 위해 다시 연락드립니다”라는 짧은 안내를 한 번 보냅니다.
- 추가 5영업일 후: “응답이 없어 티켓을 종료하지만, 같은 티켓으로 다시 연락하면 이어서 돕겠다”고 안내합니다.
- 장애·보안·계약 이슈는 고객 응답이 없어도 내부 오너에게 종료 가능 여부를 확인합니다.

## A-6. 기능 요청을 받았을 때의 상세 절차

### A-6-1. 요청을 받는 사람의 역할

요청을 받은 사람은 ‘만들어 주겠다’고 약속하는 사람이 아니라, 고객 문제를 정확히 기록해 제품팀이 판단할 수 있게 돕는 사람입니다.

아래 질문을 사용합니다.

```text
1. 이 기능이 없어서 지금 어떤 일을 못 하고 있나요?
2. 현재는 어떻게 우회하고 있나요?
3. 이 문제가 발생하는 사용자는 몇 명인가요?
4. 언제까지 해결되어야 하나요? 그 날짜가 중요한 이유는 무엇인가요?
5. 비슷한 기능 중 사용 중인 것이 있나요?
```

### A-6-2. 제품 요청 티켓 작성 예시

```text
제목: [A사] 월별 상담 건수를 CSV로 내보내고 싶다는 요청
문제: 운영팀이 월말 보고를 위해 상담 데이터를 수작업으로 복사하고 있음
영향 사용자: 고객사 운영 담당자 약 3명
현재 대안: 화면 필터 후 수작업 복사
필요 시점: 매월 말일, 7월부터 사용 희망
확정 약속 여부: 없음. 검토 후 안내 예정이라고 고객에게 전달함
관련 고객 대화: [링크]
```

### A-6-3. 제품팀의 판단 결과를 고객에게 알리는 법

| 판단 | 고객에게 말하는 방법 |
| --- | --- |
| 계획됨 | “현재 [분기/월]에 검토·개발을 계획하고 있습니다. 다만 일정은 변경될 수 있으며 확정 시 안내드리겠습니다.” |
| 검토 중 | “사용 사례를 바탕으로 적합성을 검토 중입니다. [날짜]까지 검토 결과를 안내드리겠습니다.” |
| 보류 | “현재 더 많은 고객에게 영향을 주는 과제를 우선 진행하고 있습니다. 요청은 기록해 두었고 우선순위가 바뀌면 안내드리겠습니다.” |
| 제외 | “현재 제품 방향과 맞지 않아 제공하기 어렵습니다. 대신 [대안 기능/방법]을 안내드릴 수 있습니다.” |

## A-7. 개발 업무를 처음 수행하는 사람을 위한 절차

### A-7-1. 작업 시작 전 확인 순서

1. Jira 티켓의 완료 기준을 읽습니다.
2. 같은 기능의 화면·API·데이터가 이미 있는지 코드와 문서를 검색합니다.
3. 어떤 사용자가 영향을 받는지, 관리자와 일반 사용자의 권한 차이가 있는지 확인합니다.
4. 고객 데이터가 아닌 테스트 계정·테스트 데이터에서만 개발·검증합니다.
5. 데이터베이스 변경이면 팀 리드 또는 데이터 오너에게 설계 검토를 요청합니다.

### A-7-2. 브랜치와 PR 만들기

1. `main` 최신 상태에서 새 브랜치를 만듭니다. 예: `feature/export-csv`, `fix/login-error`.
2. 한 PR에는 가능한 한 한 가지 목적의 변경만 넣습니다.
3. 변경 후 린트·타입 검사·테스트를 실행합니다.
4. PR 설명에 아래 양식을 사용합니다.

```text
## 문제
무엇이 불편하거나 잘못되었는가?

## 변경 내용
무엇을 바꾸었는가?

## 확인 방법
다른 사람이 어떻게 재현·확인할 수 있는가?

## 영향과 위험
어떤 사용자·데이터·권한에 영향이 있는가? 롤백은 어떻게 하는가?

## 화면 변경
스크린샷 또는 동영상 링크
```

### A-7-3. 코드 리뷰에서 받은 의견 처리법

- 이해하지 못한 의견은 “이 부분에서 어떤 문제가 생길 수 있는지 예시를 들어주실 수 있나요?”라고 질문합니다.
- 동의하면 수정 후, 무엇을 바꿨는지 답글로 남깁니다.
- 다른 방법이 더 낫다고 생각하면 근거와 장단점을 적습니다. 개인적 취향이 아니라 팀 규칙·성능·보안·유지보수성을 기준으로 이야기합니다.
- 리뷰 승인 없이 병합하지 않습니다. 긴급한 경우에도 리드가 지정한 긴급 배포 절차를 따릅니다.

### A-7-4. 데이터베이스를 바꿀 때 반드시 확인할 것

| 확인 항목 | 질문 |
| --- | --- |
| 기존 데이터 | 이미 저장된 행은 어떻게 되는가? |
| 되돌리기 | 오류 시 이전 구조로 되돌릴 수 있는가? |
| 권한 | 일반 사용자에게 다른 회사 데이터가 보일 가능성은 없는가? |
| 성능 | 데이터가 많아질 때 조회가 느려지지 않는가? |
| 백업 | 실수 시 복구할 방법이 있는가? |
| 순서 | 앱 배포와 마이그레이션 중 무엇을 먼저 해야 하는가? |

**멈춤 기준:** 운영 데이터에 직접 SQL을 실행해야 한다면 혼자 하지 않습니다. 데이터 오너와 최소 한 명의 검토자 승인을 받습니다.

## A-8. QA(품질 확인) 상세 체크리스트

### A-8-1. 모든 화면에서 확인할 항목

- 정상 입력을 했을 때 원하는 결과가 나오는가?
- 필수 항목을 비우면 이해 가능한 안내가 나오는가?
- 잘못된 형식(짧은 비밀번호, 잘못된 이메일 등)을 넣으면 안전하게 막는가?
- 저장 버튼을 두 번 눌러도 데이터가 중복으로 만들어지지 않는가?
- 로딩 중에 무엇을 기다리는지 보이는가?
- 오류가 나도 사용자가 다음 행동을 알 수 있는가?
- 작은 화면과 큰 화면에서 글자·버튼이 잘리지 않는가?
- 키보드만으로 주요 기능을 사용할 수 있는가?

### A-8-2. 권한 테스트 예시

새 기능이 문서 업로드 기능이라면 다음 계정으로 각각 확인합니다.

| 계정 | 기대 결과 |
| --- | --- |
| 일반 사원 | 문서 원문·업로드 메뉴에 접근할 수 없음 |
| 문서 관리자 | 문서를 업로드·교체·상태 확인 가능 |
| 다른 회사 관리자 | 우리 회사 문서나 대화에 접근할 수 없음 |
| 로그인하지 않은 사용자 | 로그인 화면으로 이동 |

### A-8-3. 버그 보고 양식

```text
제목: [환경] [기능] 어떤 문제가 발생하는가
예: [Chrome/Windows] 사용자 초대 시 저장 버튼이 계속 로딩됨

1.
2.
3.
기대한 결과:
실제 결과:
발생 빈도: 항상 / 가끔 / 한 번
증거: 스크린샷, 영상, 오류 메시지, 요청 ID
고객 영향: 없음 / 일부 사용자 / 다수 사용자
```

## A-9. 문서 관리자가 매뉴얼을 업로드·수정하는 절차

### A-9-1. 새 매뉴얼을 처음 올릴 때

1. 파일 이름을 알아보기 쉽게 정합니다. 예: `2026-07_정보보안_직원매뉴얼_v1.0.pdf`.
2. 파일 첫 페이지 또는 맨 위에 문서명, 오너, 버전, 적용일, 대상 부서를 적습니다.
3. 문서에 비밀번호·비밀키·실제 고객 개인정보가 없는지 확인합니다.
4. 관리자 화면에서 `문서 업로드` → `새 매뉴얼`을 선택합니다.
5. 제목, 문서 유형, 대상 부서, 설명, 파일을 입력하고 업로드합니다.
6. 처리 상태가 `준비됨`이 될 때까지 기다립니다.
7. 챗봇에 문서 내용과 관련된 질문 3개를 해 보고, 답변의 출처가 맞는지 확인합니다.
8. 문제가 없으면 문서 상태를 `게시`로 변경하고 관련 구성원에게 적용일을 알립니다.

### A-9-2. 기존 매뉴얼의 최신 버전을 올릴 때

1. 기존 문서가 무엇인지 검색합니다. 비슷한 제목의 새 문서를 다시 만들지 않습니다.
2. 변경 이유와 바뀐 내용을 3줄 이내로 작성합니다.
3. `최신 버전 교체`를 선택하고 새 파일을 올립니다.
4. 처리 완료 후, 변경된 항목으로 질문해 새 버전이 검색되는지 확인합니다.
5. 기존 버전은 기록용으로 남기되, 구성원이 잘못 참조하지 않도록 최신 버전 표시를 확인합니다.

### A-9-3. 문서를 게시하면 안 되는 경우

- 내용의 오너가 불명확한 초안
- 적용일이 정해지지 않은 정책
- 다른 고객사 또는 다른 조직의 기밀이 포함된 파일
- 사람 검토 없이 AI가 만든 법무·보안·인사 정책
- 이미 폐기된 절차를 설명하는 파일

### A-9-4. 챗봇 답변이 틀렸을 때

1. 질문, 잘못된 답변, 올바른 근거 문서를 기록합니다.
2. 출처 문서가 오래됐는지, 문서끼리 모순되는지, 문서가 누락됐는지 확인합니다.
3. 문서 문제라면 오너에게 수정을 요청하고 최신 버전을 업로드합니다.
4. 모델이 문서 밖의 내용을 지어낸 것처럼 보이면 관리자 답변으로 정정하고 운영 담당자에게 품질 이슈를 전달합니다.
5. 같은 문제가 반복되면 해당 질문을 검증 질문 목록에 추가합니다.

## A-10. 챗봇이 답하지 못한 질문을 사람이 답하는 절차

### A-10-1. 관리자 답변 전 확인할 것

- 질문자가 어느 회사·부서·권한에 속하는지 확인합니다.
- 최신 정책 문서 또는 담당 오너에게 근거를 확인합니다.
- 법무·인사·보안·개인정보 관련 질문은 해당 담당자의 확인 없이 단정해 답하지 않습니다.
- 개인 사정·건강·갈등 상담은 공개 채팅에 답하지 말고 지정된 비공개 창구로 안내합니다.

### A-10-2. 사람 답변 양식

```text
안녕하세요, [이름]님.

결론: [사용자가 지금 해야 할 행동]
근거: [정책명/문서명/담당 부서]
절차:
1.
2.
3.
예외 또는 주의사항: 
추가 도움이 필요하면: [담당 팀/연락 방법]
```

### A-10-3. 답변 후 해야 할 일

- 동일한 질문이 다시 나올 가능성이 있으면 FAQ 또는 매뉴얼 보완 요청을 만듭니다.
- 정책이 없어서 답변을 새로 정했다면 담당 부서가 공식 문서로 확정하도록 연결합니다.
- 답변에 고객·직원 개인정보가 들어갔다면 즉시 수정·비공개 조치하고 보안 담당자에게 알립니다.

## A-14. 판단이 어려울 때 쓰는 ‘멈춤-확인-진행’ 규칙

아래 중 하나라도 해당하면 멈추고 확인합니다.

### 멈춤이 필요한 신호

- “빨리 해달라”는 이유로 보안·승인 절차를 건너뛰라고 한다.
- 고객의 개인정보·비밀키·계약 조건을 보내 달라고 한다.
- 다른 회사의 데이터나 계정이 보인다.
- 고객에게 기능·가격·일정을 지금 확답하라고 한다.
- 운영 데이터·권한·결제를 직접 바꾸라고 한다.
- 문서와 구두 지시가 서로 다르다.

### 확인 메시지 양식

```text
이 작업은 [보안/개인정보/권한/비용/고객 약속]에 영향을 줄 수 있어 확인이 필요합니다.
제가 이해한 요청은 [한 문장]입니다.
진행 전에 [결정해야 할 항목]을 확인해 주실 수 있나요?
확인 전까지는 [안전한 임시 조치]까지만 진행하겠습니다.
```

### 진행해도 되는 경우

- 공개된 문서를 읽고 요약하는 일
- 테스트 데이터로 화면을 확인하는 일
- 이미 승인된 템플릿으로 일반 문의에 답하는 일
- 팀 리드가 완료 기준을 명확히 준 작은 범위의 업무

## A-15. 매주·매월 반복하는 점검표

### A-15-1. 구성원 주간 점검

- [ ] 내 Jira 티켓의 마감일·상태·다음 행동을 업데이트했다.

- [ ] 다음 주 우선순위를 팀 리드와 맞췄다.

### A-15-2. 팀 리드 주간 점검

- [ ] 우선순위가 P0~P3 기준에 맞는지 확인했다.

- [ ] 오래 멈춘 티켓과 고객 응답 대기 티켓을 검토했다.
- [ ] 새 입사자에게 버디·명확한 첫 업무가 배정되었는지 확인했다.
- [ ] 반복 질문을 매뉴얼·FAQ 개선 항목으로 만들었다.

### A-15-3. 문서 관리자 월간 점검

- [ ] 게시된 문서의 오너·적용일·최신 버전을 확인했다.
- [ ] 폐기된 정책을 비공개 또는 보관 처리했다.
- [ ] 챗봇이 자주 답하지 못한 질문을 검토했다.
- [ ] 상충하는 문서가 없는지 확인했다.
- [ ] 표본 질문으로 출처와 답변 정확성을 점검했다.

## A-16. 신입 사원이 알아야 할 간단한 제품 용어 사전

| 용어 | 아주 쉽게 설명 | 예시 |
| --- | --- | --- |
| 계정(Account) | 한 사람이 서비스에 로그인할 때 쓰는 정보 | 홍길동의 회사 이메일 로그인 |
| 권한(Role) | 계정으로 할 수 있는 일의 범위 | 일반 사원은 문서 업로드를 못 하고 관리자는 할 수 있음 |
| 고객사(Workspace) | 한 회사의 데이터가 모여 있는 분리된 공간 | A사와 B사는 같은 서비스여도 서로 데이터를 볼 수 없음 |
| 관리자(Admin) | 설정·문서·사용자를 관리할 수 있는 역할 | 문서 관리자, 회사 관리자 |
| 티켓(Ticket) | 해야 할 일을 기록하고 끝까지 추적하는 카드 | 고객 오류, 기능 요청, 권한 요청 |
| 에스컬레이션 | 혼자 처리하면 위험하거나 권한이 없어 담당자에게 넘기는 것 | 개인정보 삭제 요청을 개인정보 담당에게 전달 |
| 배포(Deploy) | 개발한 변경을 실제 사용자에게 적용하는 것 | 로그인 오류 수정 내용을 서비스에 반영 |
| 롤백(Rollback) | 문제가 생긴 변경을 이전 상태로 되돌리는 것 | 배포 후 오류가 나 이전 버전으로 복구 |
| 로그(Log) | 시스템에서 언제 무슨 일이 일어났는지 남긴 기록 | 로그인 실패 시간과 오류 코드 |
| 테스트 환경(Staging) | 실제 고객에게 영향 없이 미리 확인하는 공간 | 새 화면을 실제 배포 전 확인 |
| 프로덕션(Production) | 고객이 실제로 사용하는 서비스 | 고객이 로그인하는 NovaDesk 웹사이트 |
| 개인정보 | 특정 사람을 알아볼 수 있는 정보 | 이름, 이메일, 전화번호, 상담 내용 |
| 비밀키(Secret Key) | 시스템 권한을 여는 매우 중요한 비밀번호 | API 키, 서비스 역할 키 |
| SLA | 고객에게 약속한 지원·서비스 기준 | 장애 문의를 몇 시간 안에 답하는지 |

## A-17. 매뉴얼을 읽은 뒤 확인하는 연습 문제

신입 사원은 아래 상황을 버디와 함께 이야기해 봅니다. 정답을 외우는 것보다, 어떤 문서·담당자·절차를 찾아야 하는지 설명하는 것이 목표입니다.

1. 고객이 “내일 오전까지 새 기능을 만들어 달라”고 요청했다. 무엇을 먼저 말하고, 누구에게 확인해야 하는가?
2. Slack에서 API 키가 보이는 메시지를 발견했다. 첫 5분에 무엇을 해야 하는가?
3. 일반 사원이 매뉴얼 원문을 달라고 요청했다. 어떤 권한과 대안을 안내해야 하는가?
4. 고객이 본인 회사의 모든 데이터를 지워 달라고 했다. 바로 삭제해도 되는가?
5. 배포 후 로그인 오류가 발생해 고객 여러 명이 접속하지 못한다. 누가 IC가 되고, 고객에게 무엇을 알려야 하는가?

### 연습 문제 통과 기준

- “누구에게 물어본다”가 아니라 담당 역할(예: 보안 담당, PM, IC)을 말할 수 있다.
- 고객 정보와 비밀키를 받거나 공유하지 말아야 한다는 이유를 설명할 수 있다.
- 문서에 없는 상황에서는 추측하지 않고 티켓·에스컬레이션을 사용한다고 말할 수 있다.

## A-18. 매뉴얼 오너를 위한 작성 품질 기준

새 절차를 추가할 때 아래 기준을 충족해야 합니다.

1. **대상:** 누가 읽는 문서인지 명확한가?
2. **시작 신호:** 어떤 일이 생기면 이 절차를 시작하는가?
3. **순서:** 번호가 있는 행동 단계가 있는가?
4. **결과:** 어디까지 하면 완료인지 알 수 있는가?
5. **예외:** 위험하거나 판단이 어려울 때 멈출 기준이 있는가?
6. **담당:** 누구에게 질문·승인을 받아야 하는가?
7. **근거:** 관련 정책·양식·시스템 링크가 있는가?
8. **검토:** 실제 초보자가 읽고 실행해 본 뒤 모호한 표현을 고쳤는가?

이 기준을 충족하지 못한 문서는 ‘참고 자료’로만 게시하고, 필수 업무 절차로 지정하지 않습니다.
